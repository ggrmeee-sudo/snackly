# Безопасность и соответствие курсовым критериям (Snackly)

Документ для отчёта: аудит клиента и сервера, TLS, контроль доступа.

## TLS / SSL-сертификаты

- **GitHub Pages** и **Render** выдают посетителям **HTTPS** и валидный сертификат автоматически; в репозитории сертификаты не хранятся (так и должно быть: приватный ключ не в git).
- Сервер **Express** при деплое за прокси получает заголовок `X-Forwarded-Proto`; для не-localhost включён **редирект 301** с `http://` на `https://` (см. `backend/src/server.js`).

## Техническая оптимизация и маршрутизация

- **`frontend/public/robots.txt`** — правила для роботов и ссылка на sitemap.
- **`frontend/public/sitemap.xml`** — карта сайта (канонический URL GitHub Pages).
- **`frontend/public/_redirects`** — правило SPA для Netlify-совместимых хостингов.
- После сборки Vite копирует **`index.html` → `404.html`** для корректной выдачи SPA на GitHub Pages при ошибочных путях.
- **Express**: отдача статики, затем **fallback** на `index.html` для маршрутов без расширения файла (не подменяет 404 для `.js`, `.css` и т.д.).

## Управление состоянием и клиентский код

- Данные корзины, сессии и заказов хранятся в **`localStorage`**; реактивный снимок для UI — **`clientState`** (Vue 3 `reactive`) в `frontend/src/clientStore.js`.
- **Vue 3** используется как **движок реактивного UI** для суммы в плашке корзины (`frontend/src/reactiveCartPill.js`) без изменения CSS/разметки страницы.

## Асинхронность и DOM

- Загрузка и проверка доступности API: **`fetch`**, **`async`/`await`**, **`AbortController`** — `frontend/src/apiProbe.js`; событие `snackly-connectivity` для отладки.
- Остальной интерфейс по-прежнему управляет **DOM** через модули (`auth.js`, `shop.js`, `cart.js`, …).

## Аудит: контроль доступа

| Угроза | Оценка |
|--------|--------|
| Обход «входа» | Сессия только в **localStorage**; пользователь может изменить данные в DevTools. Для учебного проекта приемлемо; для продакшена нужны серверные сессии/JWT и проверка на API. |
| Доступ к чужим заказам | Заказы в `localStorage` связаны с email сессии на клиенте; серверной изоляции нет. |
| XSS | Ввод в формы в основном уходит в **`textContent`** или экранируется данными из каталога; фрагменты **`innerHTML`** собираются из контролируемых полей (даты, суммы). Риск при появлении «сырого» HTML из пользовательского ввода — избегать `innerHTML` для таких данных. |

## Аудит: сервер

- Раздача **статики** и **`GET /api/health`**; тела запросов не парсятся, отдельного CORS для сторонних origin нет.
- Заголовки: **`X-Content-Type-Options`**, **`X-Frame-Options`**, **`Referrer-Policy`**, **`Permissions-Policy`** (см. `server.js`).

## Рекомендации при усложнении проекта

- Вынести авторизацию и заказы на сервер с проверкой прав.
- Подключить **Content-Security-Policy** после подбора политики под шрифты и скрипты.
- Ограничить **rate limiting** на API при появлении записи данных с клиента.
